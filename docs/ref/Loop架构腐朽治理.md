最近看到了一个词：架构腐朽。

这个词有点重，但放到真实系统里，又挺准确。

很多老系统让人头疼，代码难看反倒还排在后面。更麻烦的是，删除变成了一件没人能轻易拍板的事。

字段先留着，接口继续开着，旧配置也暂时不动，某段明显别扭的逻辑同样先绕过去。大家知道它有问题，只是没人能说清它还被谁依赖，删掉会不会牵出一条很远的业务链路。

放到 AI Agent 时代，这个问题会更明显。Agent 能更快写代码，也能更快把临时处理、旧误解和模糊边界沉积到系统里。Loop Engineering 如果只被理解成“让 Agent 一直跑”，它未必在治理复杂度，反而可能把复杂度自动化。

我的看法会更保守一点：

Loop Engineering 放在架构治理里，重点不在“多跑几轮”。我更关心的是，它能不能让系统持续发现、验证和减少那些越来越难处理的历史负担。

这也接上了我们今年一直在梳理的那条线：Prompt 解决“怎么说清楚”，Context 解决“该给它看什么”，Harness 解决“怎么让它跑起来”，Loop 开始关心“这类事怎么持续发生”。到这里再谈架构腐朽，它就从老系统问题，变成了 Agent 时代的运行时问题。

它变成了一个更具体的问题：能不能给系统做一套持续清理自己的机制。

先说结论
架构腐朽有一个很朴素的信号：删除风险开始大于保留成本。

一旦系统进入这个状态，团队会自然选择“只加不减”：旧字段旁边加新字段，旧服务外面包适配层，旧规则旁边补例外。每一次选择都有理由，长期看却会让系统越来越难理解。

Loop Engineering 的价值，是把这类问题从“偶尔专项治理”变成 小而持续的反馈循环。

更具体一点，一个面向架构排熵的 Loop 可以先做五件事：

• 定期发现新增复杂度；
• 把 证据 写清楚；
• 只在低风险范围内尝试修复；
• 用独立验证挡住自写自审；
• 在没有进展或边界不清时 停下来。
这几个动作不炫，但很实用。

这里还有一个前提：不是所有事情都值得做成 Loop。

一个任务至少要过三道小门槛：重复发生，值得自动化；结果可复核，知道什么叫通过；收益足够大，扛得住模型、工具和验证的成本。缺一项，先别急着上循环，手动处理或者写个普通脚本可能更稳。

先看删除为什么变贵
Hyrum 定律说得很直接：只要用户足够多，系统所有可观察行为都会被某个人依赖。

这句话在老系统里很常见。文档没有承诺返回顺序，但调用方已经依赖顺序；异常文案本来只是提示，几年后变成了监控规则；一个本来临时加的字段，后来被报表、脚本、客服流程一起用上。

这时问题已经超出了代码质量。

更贴近现场的说法是，系统失去了影响范围推理能力。团队不知道一个改动会碰到谁，也不知道一次删除会不会把远处的链路拉倒。于是保留变得便宜，删除变得昂贵。

Lehman 的软件演化定律也提醒过这件事：真实使用中的系统会持续变化；如果没有专门投入去控制复杂度，复杂度会增长，质量会下降。

所以健康架构并不是“永远保持原样”。

它更像一种代谢能力：能吸收新需求，也能排出旧负担；能引入新规则，也能退役旧规则。

Agent 会放大这件事
过去，腐朽发生得慢一点。

工程师写代码、提 PR、等 CI、被 reviewer 问几句。这个过程有摩擦。摩擦有时候烦，但也会迫使团队把理由、边界和验证说清楚。

Agent 加进来以后，摩擦变小了。它可以很快补文档、补测试、改配置，也可以把一段没有完全验证的结果总结得像已经闭环。

短期看，这很舒服。长期看，代码库里可能多出更多“看起来有道理，但没有进入团队共同理解”的东西。

Naur 在《Programming as Theory Building》里有个重要观点：程序不只是代码，还包括维护者头脑里的系统理论。代码是理论的投影，不是理论本身。

模型可以生成代码，但它不天然继承这套理论。会话结束，很多判断就没了；上下文被裁剪，很多理由就断了；模型升级，原来稳定的写法也可能变化。

所以，Agent 时代的架构治理要先问一个问题：

哪些理由、约束、验证和状态，不能只放在模型上下文里，而要回到系统本身？

到这里，Loop Engineering 才和架构治理接上。

这和前面梳理 Harness、Self-Harness、Environment 时的思路是一致的：Agent 能不能变强，除了模型，还要看工作现场能不能留下证据、给出反馈、允许回滚。今天只是把这个问题换到架构治理里看。

一个排熵 Loop 怎么设计
第一版排熵 Loop 可以设计得很窄。

它不负责“大规模自动重构”，也不直接碰生产配置。它先负责发现新增问题、生成证据、尝试低风险修复，并把无法判断的部分交还给人。

一个可用的最小结构大概是这样：

触发 -> 读取状态 -> 扫描证据 -> 低风险修复 -> 独立验证 -> 写回账本

这里先看边界，再看 Agent 有多聪明。

如果状态只留在聊天记录里，下一轮就像重新入职。如果证据只剩一句“已修复”，人接手时还是要重来。如果执行者和验证者是同一个角色，质量判断很容易偏乐观。

所以 状态账本 要当作核心资产。它可以很朴素，就是一个 Markdown、issue、看板或数据库记录。关键是每一轮都留下四类信息：看到了什么，做了什么，验证了什么，还剩什么风险。

三个容易踩的坑
From Agent Loops to Structured Graphs 那篇论文里有个提醒很有用：普通 Agent Loop 的问题，很多时候不在模型能力，而在控制结构太隐。

图片
图 3：普通 Agent Loop vs 受控排熵 Loop
第一个坑，是步骤依赖藏在上下文里。

“先改代码，再跑测试，再根据失败结果修复”，在人脑里很自然。但系统如果没有显式依赖关系，就不知道哪一步要等哪一步完成，也不知道某一步失败后，后续动作是否还有效。

第二个坑，是恢复循环没有边界。

修 CI 时最常见：失败后再改，再跑，再失败，再换方案。每一轮都像有进展，但如果没有轮数上限、无进展检测和升级条件，它会在同一个坑里绕很久。

第三个坑，是执行历史被摘要改写。

聊天上下文会压缩，运行日志会被总结，失败原因可能最后只剩一句“已处理”。如果真实命令、diff、测试输出和失败路径没有写回系统，后面看到的只是历史摘要，不是完整历史。

这三个坑对应三条改法：

• 依赖要显式化；
• 恢复要有上限；
• 历史要写回系统。
这比“让 Agent 再聪明一点”更可靠。

先从四个小场景开始
第一步先别急着做“自动架构师”。更现实的起点，是几个目标窄、验证便宜、失败可回滚的小 Loop。

1. 架构规则基线

扫描循环依赖、跨层调用、违规 import。第一版先不背历史包袱，目标放在阻止新增违规上。baseline 只减不增，系统就开始有排熵方向。

2. 废弃路径清单

定期读取访问日志、代码搜索、文档引用和告警规则，把旧接口、旧配置、旧字段分成几类：仍有流量、无流量但有引用、仅文档残留、可以进入删除评审。

这里先出清单，不直接删除。删除涉及兼容性和责任边界，仍然要人来判断。

3. 文档漂移检查

把 README、CI 配置、运行脚本和实际命令放在一起对账。命令还能不能跑，环境变量是否还存在，文档里的功能描述是否过期，这些都很适合交给 Loop 做第一轮核验。

4. 检查器健康度

治理系统也会老。一个检查器如果 30 天从未失败，可能是系统很健康，也可能是检查器已经失效。可以定期检查：规则是否长期被跳过，baseline 是否增长，suppress 是否没有 owner，检查耗时是否让团队开始绕过它。

这些场景都不宏大，但能落地。

它们的共同点是：读多、改少、证据清楚，最后结果容易复核。

人还在现场
Loop Engineering 容易被讲成“人终于可以退后一步”。这句话要加边界。

在架构治理里，很多问题会落到几个很具体的判断上：现在值不值得改，兼容成本由谁承担，旧行为是不是已经变成事实契约。

这些判断不能交给循环。

Loop 更适合做前半段：持续发现问题，把低风险修复推进到有证据的候选状态，把失败和例外写回系统。最后的取舍，仍然要回到人。

生产版 Loop 还有一个容易被低估的设计目标：会停。

成本上限、轮数上限、无进展检测、人工升级，都要提前写清楚。一个不会停的 Loop，和一个不敢删的系统，其实有相同的问题：都失去了退出机制。

写在最后
架构腐朽过去更多被当成老系统问题。Agent 进入研发现场后，它开始变成运行时问题。

代码生成变快，流程执行变快，反馈循环也变快。系统可以更快变好，也可以更快沉积没人理解的历史。

所以我现在看 Loop Engineering，不太愿意只看“提示词之后是什么”。

更值得盯的是：能不能把过去靠人记、靠人催、靠专项清理的架构治理动作，变成小而稳定的排熵循环。

好的 Loop 不会替系统做最终判断。它会让问题更早出现，让证据更容易复核，让旧负担更容易进入退场流程。

这已经很有价值。

只要系统还能发现自己哪里正在变错，腐朽就不会无声完成。
